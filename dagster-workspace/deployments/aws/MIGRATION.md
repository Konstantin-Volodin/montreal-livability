# Migration - Dagster state: sqlite-on-EFS → Aurora Serverless v2 postgres

Planned, not yet implemented. Execute top to bottom; delete this file when done.

## Why

- `efs_state()` shuttles a sqlite `$DAGSTER_HOME` to/from EFS around each run - hand-rolled durability (partial copies, concurrent-writer fragility), unreachable from outside the VPC.
- Aurora Serverless v2 at **min 0 ACU** - auto-pauses between monthly runs, storage-only idle cost, and a laptop `dagster-webserver` can point at it directly.
- Model unchanged: daemon-less, EventBridge → one-shot Fargate, `batch.py` staging, `run.py`.
- Old EFS history not migrated - first run is a full recompute; the RETAIN'd filesystem is orphaned on deploy, delete manually later.

Verified against installed packages:

- aws-cdk-lib 2.257.0 accepts `serverless_v2_min_capacity=0` + `serverless_v2_auto_pause_duration`, validated at **synth** against the engine version.
- dagster 1.13.5 `storage: postgres` autocreates tables on first connect - no `dagster instance migrate` step.
- ECS-bound `ecs.Secret`s auto-grant `secretsmanager:GetSecretValue` to the execution role - no manual policy.
- `defs/resources/state.py` imported only by `batch.py` - safe to delete.

# State - Aurora Serverless v2 postgres for the Dagster instance; 0 ACU between
# monthly runs (resume ~15s, the first connection just waits it out).
db = rds.DatabaseCluster(
    self, "DagsterDb",
    engine=rds.DatabaseClusterEngine.aurora_postgres(version=rds.AuroraPostgresEngineVersion.VER_17_5),
    writer=rds.ClusterInstance.serverless_v2("writer"),  # publicly accessible by default in PUBLIC subnets
    serverless_v2_min_capacity=0, serverless_v2_max_capacity=0.5,
    serverless_v2_auto_pause_duration=cdk.Duration.minutes(15),
    default_database_name="dagster",
    vpc=vpc, vpc_subnets=ec2.SubnetSelection(subnet_type=ec2.SubnetType.PUBLIC),
)
db.connections.allow_default_port_from(task_sg)
if home_ip: db.connections.allow_default_port_from(ec2.Peer.ipv4(home_ip), "dagster-webserver from home")

2. `python run.py` + `aws logs tail /ecs/livability --follow` - slow first connection (resume from 0), table autocreation, every cache gate logs "no prior materialization on this instance", full recompute, `quality/<run>.json` in S3
3. second `python run.py` - the real assertion: `address_r6` partitions enumerate from postgres, gates log "inputs unchanged … reusing prior snapshot"
4. laptop webserver per the README section - both runs visible
5. RDS console - 0 ACU ~15 min after the run ends

## Risks

- **0-ACU gate** - CDK fails at synth if the engine version lacks auto-pause - cheap to catch, bump the 17.x version. Worst case `min_capacity=0.5` ≈ $45/mo standing - rethink if hit.
- **Resume latency** - ~15-30s; the endpoint holds connections during resume, psycopg2 has no connect timeout - handle by doing nothing. If flaky: `params: {connect_timeout: 60}` under `postgres_db`.
- **Public DB** - exposure = generated Secrets Manager password behind SG (task SG + `home_ip` /32 only). Droppable in two lines.
- **Standing cost no longer zero** - paused cluster bills storage/backups (cents at this size).
- **uv.lock drift** - `--frozen` build fails if `uv lock` is skipped after the dep add.
